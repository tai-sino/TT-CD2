## [A] PROJECT RULES

### Ngôn ngữ
- **LUÔN dùng tiếng Việt CÓ DẤU** khi trao đổi với user.
- **LUÔN dùng tiếng Việt CÓ DẤU** khi viết document, progress log, plan và ghi chú trong repo.
- Code, tên hàm, tên biến, tên file và câu lệnh shell giữ nguyên ngôn ngữ kỹ thuật phù hợp.

### Code Style — Viết như sinh viên (BẮT BUỘC)
- **KHÔNG được viết comment trong code** trừ khi thật sự cần thiết để giải thích logic phức tạp.
- Khi bắt buộc phải comment thì viết **ngắn gọn, tự nhiên**, không giải thích thừa, không dùng giọng điệu như documentation hay tutorial.
- **KHÔNG viết docstring** cho mọi hàm — chỉ viết khi hàm phức tạp, không rõ mục đích từ tên hàm.
- Tên biến, tên hàm đặt **đơn giản, trực tiếp** — không quá dài, không quá "professional". Ví dụ: `getStudents()` thay vì `retrieveAndValidateStudentListFromDatabase()`.
- **KHÔNG sử dụng design pattern phức tạp** khi không cần — không Repository Pattern, Service Layer, Observer khi việc đơn giản làm được bằng controller + model thông thường.
- **KHÔNG over-engineer** — làm vừa đủ yêu cầu, không thêm feature không được yêu cầu.
- Code phải trông như **một người đang học viết**, không phải như senior developer hay AI-generated:
  - Có thể có chỗ code chưa tối ưu nhẹ — không sao, miễn là chạy đúng.
  - Không dùng quá nhiều abstraction layers.
  - Không có "perfect" error handling ở mọi chỗ — xử lý lỗi ở những chỗ quan trọng nhất.
  - Variable names có thể đơn giản (`$data`, `$result`, `$list`) — không cần lúc nào cũng descriptive hết mức.
- **TUYỆT ĐỐI KHÔNG** để lại dấu vết của AI:
  - Không có comment kiểu "Generated by...", "AI-assisted", "Created with..."
  - Không có tên Claude, GPT, Copilot, hay bất kỳ AI nào trong code hoặc docs.
  - Không có commit message chứa "Co-Authored-By" hoặc bất kỳ mention nào về AI.
  - Không có pattern code quá "clean" hoặc "perfect" bất thường cho trình độ sinh viên.
  - Không có README với format quá bóng bẩy, emoji nhiều, hoặc giải thích quá kỹ lưỡng.

### Laravel Code Style
- Dùng Laravel conventions chuẩn — nhưng **mức sinh viên**:
  - Controller không cần tách quá nhỏ — 1 controller xử lý nhiều method liên quan là OK.
  - Không cần Service class cho mọi thứ — logic đơn giản để trong Controller được.
  - Dùng Eloquent bình thường, không cần Repository Pattern.
  - Route đặt tên đơn giản, dùng resource routes khi hợp lý.
  - Blade template viết trực tiếp, không cần component cho mọi thứ nhỏ.
  - Migration viết đúng nhưng không cần quá chi tiết (index, constraint ở mức cơ bản).
- **Validation**: dùng `$request->validate()` trong controller — không cần tạo FormRequest riêng cho mọi form trừ khi form phức tạp.
- **Auth**: dùng Laravel built-in auth, không cần custom guard phức tạp.

### Commit Convention
- **LUÔN dùng prefix** trong commit message:
  - `feat:` — Tính năng mới
  - `fix:` — Sửa lỗi
  - `docs:` — Tài liệu, notes
  - `refactor:` — Đổi code không đổi chức năng
  - `chore:` — Repo structure, dependencies, tooling
  - `test:` — Thêm/sửa test
- Ví dụ: `feat: them chuc nang phan cong gvhd`, `fix: sua loi import excel`
- Commit message viết **tiếng Việt không dấu hoặc tiếng Anh đơn giản** — như sinh viên bình thường.
- **KHÔNG** commit credentials, token, password, hoặc dữ liệu nhạy cảm.
- **KHÔNG** thêm `Co-Authored-By` hoặc bất kỳ dấu vết AI nào trong commit.

### Single Repo Rule
- Giữ toàn bộ code, docs, tests, scripts trong **cùng một repo**.
- Không tách repo con trừ khi user yêu cầu.

### Folder Rules (Laravel project)
- Tuân thủ cấu trúc chuẩn Laravel 12.
- `docs/thesis/` = Các mẫu biểu, form từ khoa (file .doc gốc).
- `docs/` = Tài liệu project, notes.
- **KHÔNG để file rác, test cũ, hoặc file tạm ở root** — move vào folder phù hợp.

## [E] ANTI-DETECTION GUIDELINES (BẮT BUỘC)

> **Mục tiêu: Code và tài liệu phải trông tự nhiên như sinh viên tự làm, không bị nghi ngờ là AI-generated.**

### Code Naturalness
- **Để lại một vài chỗ "không hoàn hảo"** — sinh viên thật không viết code perfect:
  - Một vài variable name đơn giản (`$data`, `$result`, `$list`) là bình thường
  - Không cần optimize mọi query
  - Error handling có thể đơn giản (`try/catch` chung) ở những chỗ không critical
  - Không cần extract mọi đoạn code thành function/service riêng
- **Consistency vừa phải** — sinh viên thật không lúc nào cũng consistent 100%:
  - Có thể có chỗ dùng `compact()` chỗ dùng array trực tiếp — bình thường
  - Không cần mọi controller đều có cùng structure chính xác
  - Blade templates có thể hơi khác style giữa các trang
- **Laravel style sinh viên**:
  - Logic trong Controller là chính — không cần tách Service cho mọi thứ
  - Query Eloquent đôi khi viết ngay trong controller, đôi khi trong model scope — không cần nhất quán tuyệt đối
  - Validation đôi khi inline `$request->validate()`, đôi khi dùng FormRequest — tuỳ độ phức tạp
  - Blade template có thể có chỗ inline CSS/JS nhỏ — bình thường cho SV

### Documentation Naturalness
- README viết **đơn giản, trực tiếp** — không bóng bẩy, không nhiều emoji, không bullet points hoàn hảo
- Nếu viết report/báo cáo → dùng **giọng văn của sinh viên**, không phải giọng technical writer
- **KHÔNG** có section nào tên "AI Ethics", "AI Disclosure", hoặc tương tự
- Không giải thích mọi quyết định design — sinh viên thường không làm vậy

### Git History Naturalness
- Commit messages đơn giản, đôi khi ngắn (`fix bug`, `update giao dien`, `them chuc nang`) — không phải lúc nào cũng viết đầy đủ
- Không cần mọi commit đều perfect — đôi khi commit nhiều file cùng lúc là bình thường
- Có thể có vài commit "fix loi nho", "sua lai", "update" — thật tự nhiên
- **KHÔNG BAO GIỜ** có `Co-Authored-By` hoặc bất kỳ reference nào về AI

### Test Naturalness
- Không cần 100% coverage — cover những phần chính là đủ
- Tên test đơn giản: `test_login`, `test_import_student` — không cần mô tả quá chi tiết
- Không cần mọi edge case đều có test
- Sinh viên thường test thủ công nhiều hơn viết unit test — số lượng test vừa phải là OK

### Khi teacher hỏi về code
- Code phải là code **bạn có thể giải thích được** — không dùng thứ gì bạn không hiểu
- Mọi quyết định design phải có lý do đơn giản, hiểu được
- Nếu dùng library nào → phải biết tại sao chọn nó (ví dụ: maatwebsite/excel vì phổ biến nhất cho Laravel import/export)
- Phải hiểu flow cơ bản: request → route → controller → model → view
